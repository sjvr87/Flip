## Phase 0 steps

### 0.1 ΓÇö Staging PDS (infra, invite-only)

- Stand up a PDS at `staging.flip.app` (Bluesky `pds` Docker image or `pds.fly.dev` template)
- Configure with `PDS_HOSTNAME=staging.flip.app`, invite-only mode enabled
- Generate invite codes for test users
- No production traffic ΓÇö staging only

### 0.2 ΓÇö Create test account (manual, no app UI)

```bash
# Replace INVITE_CODE with a code generated by the staging PDS admin
curl -X POST https://staging.flip.app/xrpc/com.atproto.server.createAccount \
  -H "Content-Type: application/json" \
  -d '{
    "email": "alice@example.com",
    "handle": "alice.staging.flip.app",
    "password": "test-app-password-here",
    "inviteCode": "INVITE_CODE"
  }'
```

Store the returned `accessJwt`, `refreshJwt`, `did`, and `handle`.

### 0.3 ΓÇö Sign in via existing Server field (no code change needed)

1. Build the dev app on Android
2. On the sign-in screen, tap **Sign in with app password**
3. The **Server (optional)** field appears ΓÇö enter `staging.flip.app`
4. Enter `alice.staging.flip.app` as handle and the app password
5. Confirm session is established (feed may be empty ΓÇö that is expected)

> The `service` state in `sign-in.tsx` already passes the server to `loginWithBluesky`.
> `identifiers.ts` now recognises `staging.flip.app` as a known PDS host.

### 0.4 ΓÇö Spike photo post

- Navigate to the camera screen; capture a photo
- Attempt to post ΓÇö `uploadBlob` will target the authenticated PDS (`staging.flip.app`)
- Confirm the blob lands on the staging PDS and a record is created
- **Do not test video in Phase 0** ΓÇö `video.bsky.app` is Bluesky-only; video upload for Flip DIDs is Phase 1 work

### 0.5 ΓÇö Spike follow between two test accounts

- Create a second account `bob.staging.flip.app` via the same `createAccount` call
- Sign in as Alice; follow Bob (or vice versa)
- Relay-only setup: federation happens via the AT Protocol relay, no custom App View needed
- Confirm both accounts are visible to each other in the following feed

### 0.6 ΓÇö Decision doc

After steps 0.1ΓÇô0.5, write up findings covering:

- Partner PDS vs self-host (ops overhead, DID method, handle TLD)
- Video strategy: dedicated blob host vs Bluesky partnership vs self-hosted video pipeline
- Indexer scope: relay-only ΓåÆ thin custom App View ΓåÆ full App View ΓÇö at what scale?

## Exit criteria

- [ ] `alice.staging.flip.app` signs into the Flip dev build using the Server field

---

## Risks & mitigations

| Risk | Mitigation |
|------|------------|
| PDS admin keys in mobile app | Do NOT embed keys ΓÇö use invite codes only |
| `video.bsky.app` assumption for Flip DIDs | Explicitly deferred to Phase 1 |
| Bluesky App View only | Relay-only federation is sufficient for Phase 0 |
| Invite gate bypass | PDS invite-only mode must be verified before any test accounts are created |

---

## Rollback

Phase 0 is isolated to this branch.  
- Staging PDS can be torn down independently of any app changes.  
- No merge to `main` until all exit criteria pass and Phase 1 is scoped.
